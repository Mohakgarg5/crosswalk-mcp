# Crosswalk M13 Implementation Plan — Per-ATS selector packs (apply_application v1.4)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship v1.4.0 with per-ATS selector packs. The `Browser.fillForm` method gains an optional `ats` hint; when present, ATS-specific selector candidates are tried BEFORE the existing generic candidates. `apply_application` derives the hint from the application's company → ATS in the registry (no URL-pattern guessing) and passes it through. The big concrete wins: Workable's camelCase form names (`firstName`, `lastName`, `phoneNumber`, `resumeFile`) and Ashby's `data-testid`-based attributes both become reliably matched. Greenhouse/Lever continue to work via existing generic selectors.

**Architecture:**
- `Browser.fillForm(url, fields, opts?)` accepts an optional `{ ats?: string }` param. Backward-compatible: existing callers (none other than `apply_application`) need no change. The interface signature change is what makes per-ATS routing possible without polluting `fields[]` with ATS metadata.
- A new `ATS_SELECTORS: Record<string, Partial<Record<StaticKind, string[]>>>` overlay sits next to the existing `SELECTORS` table in `playwright.ts`. Each key is an ATS slug; the value maps fill kinds to candidate selector lists. Overlay candidates are tried FIRST (prepended) to the generic SELECTORS list — so generic still wins when overlay misses, and overlay wins on its strong signals.
- `apply_application` looks up the ATS by `getJob(db, app.jobId)` → `getCompany(db, job.companyId).ats`. That's reusing the same DB-lookup pattern M12 already established for sampling job/applicant context. Passes the slug into `browser.fillForm(url, fields, { ats })`. Surfaces it as `detectedAts: string | null` on the result so users can verify which pack ran.
- v1.4 ships meaningful packs for **Workable** and **Ashby** (where the wins are biggest). **Greenhouse** and **Lever** packs are added as empty placeholders to anchor the architecture — when v1.4.x identifies new selectors, they drop in without further refactor. **Workday** stays out (its custom-widget framework needs a separate v1.5 milestone).

**Tech Stack:** Same as M12. No new dependencies.

**M13 ships:**
- `Browser.fillForm` signature gains optional `{ ats?: string }` opts param
- `ATS_SELECTORS` overlay structure in `playwright.ts`
- Workable pack: `firstName`, `lastName`, `phoneNumber`, `resumeFile` selectors
- Ashby pack: `data-testid`-based selectors for email, phone, resume_file
- Greenhouse and Lever packs as empty placeholders (architecture present, no new selectors yet)
- `apply_application` reads ATS from DB and passes to fillForm; surfaces `detectedAts` on result
- README + USER_GUIDE updates noting per-ATS coverage
- Version bump 1.3.0 → 1.4.0; tag + push

**Out of M13 (deferred):**
- **Workday widget support** (`data-automation-id` selectors). Workday's Lightning components warrant a focused milestone. v1.5/M14.
- Per-ATS packs for SmartRecruiters / BambooHR / Recruitee / Personio / iCIMS — adding the 4 main ATS packs is enough scope for one milestone; the long tail follows. v1.5+/community PRs.
- URL-pattern fallback ATS detection (for forms whose company isn't in the DB). The DB-lookup path covers our actual flow today; URL patterns would be additive defense in depth, not currently load-bearing.
- Form selector packs for non-fill kinds (selects, radios, checkboxes). v1.5+ scope.

```ts
type StaticKind = Exclude<FillField['kind'], 'text_by_name'>;

/** ATS-specific selector overlays. Tried BEFORE generic candidates. */
const ATS_SELECTORS: Record<string, Partial<Record<StaticKind, string[]>>> = {
  workable: {
    first_name: ['input[name="firstName"]'],
    last_name: ['input[name="lastName"]'],
    phone: ['input[name="phoneNumber"]'],
    resume_file: ['input[name="resumeFile"]', 'input[name="cv"]']
  },
  ashby: {
    email: ['input[data-testid*="email" i]'],
    phone: ['input[data-testid*="phone" i]'],
    resume_file: ['input[data-testid*="resume" i]']
  },
  greenhouse: {},
  lever: {}
};

function selectorsForKind(kind: StaticKind, ats: string | undefined): string[] {
  const overlay = ats ? ATS_SELECTORS[ats]?.[kind] : undefined;
  return overlay && overlay.length > 0
    ? [...overlay, ...SELECTORS[kind]]
    : SELECTORS[kind];
}
```
