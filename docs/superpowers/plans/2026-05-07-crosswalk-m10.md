# Crosswalk M10 Implementation Plan — Cover-letter fill (apply_application v1.1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship v1.1.0 by extending `apply_application` to fill cover-letter inputs — both the **file upload** (separate `<input type="file">` for cover letter, common on Greenhouse/Lever) and the **inline textarea** (`textarea[name="job_application[cover_letter]"]` and friends). The tool now materializes both the tailored resume DOCX *and* the cover-letter DOCX, attempts to match each to the appropriate input, and reports fills/skips. Submission semantics unchanged: still "review then click Submit yourself."

**Architecture:** Add 2 new variants to `FillField` (`cover_letter_file`, `cover_letter_text`). Extend the `SELECTORS` table in `playwright.ts` with candidate selectors for each. Add a sibling helper `writeCoverLetterDocxToTemp` next to the existing `writeResumeDocxToTemp` (same shape, different filename prefix). Update `apply_application.ts` to push both fields when `app.coverLetterMd` is non-empty. No new tool; tool count stays at 18.

**Tech Stack:** Same as M9. No new dependencies.

**M10 ships:**
- `FillField` gains `cover_letter_file` and `cover_letter_text` variants
- `SELECTORS` gains entries for both kinds (covering Greenhouse/Lever/Ashby patterns)
- `writeCoverLetterDocxToTemp(md, applicationId)` helper
- `apply_application` materializes the cover-letter DOCX and pushes both file + text fields
- Updated apply_application result type (no breaking change — `filled` / `skipped` arrays just gain new possible string values)
- README + USER_GUIDE updates noting the expanded fill capabilities
- Version bump 1.0.0 → 1.1.0; tagged release v1.1.0

**Out of M10 (deferred):**
- **answerPack-driven textarea fills** (v1.0 roadmap had this for v1.1; deferring to v1.2/M11). Reason: answerPack keys are arbitrary text from sampling; matching them to form field `name` attributes requires either (a) a normalization layer or (b) a form-introspection-then-sample pipeline. Either is its own milestone. The current v1.1 lands a clean win — cover letter — without that complexity.
- Cover-letter selectors for ATSes outside Greenhouse/Lever/Ashby (Workday, iCIMS) — those use custom widget frameworks; tackled per-ATS later.
- Auto-generation of cover letter when `app.coverLetterMd` is empty (`draft_application` already populates it; if a user creates an Application without a cover letter, that's their choice — apply_application skips both cover-letter fields).
- v2 still owns: actual submit click, elicitation gates, captcha, multi-step wizards.

```ts
import * as fs from 'node:fs/promises';
import * as path from 'node:path';
import * as os from 'node:os';
import { mdToDocxBuffer } from '../../exporters/docx.ts';

async function writeDocxToTemp(md: string, prefix: string, applicationId: string): Promise<string> {
  const buf = await mdToDocxBuffer(md);
  const filename = `crosswalk-${prefix}-${applicationId}-${Date.now()}.docx`;
  const filepath = path.join(os.tmpdir(), filename);
  await fs.writeFile(filepath, buf);
  return filepath;
}

/**
 * Write a tailored resume Markdown to a temp DOCX file and return its absolute path.
 * The file lives in os.tmpdir() and is left for the OS to reap.
 */
export async function writeResumeDocxToTemp(resumeMd: string, applicationId: string): Promise<string> {
  return writeDocxToTemp(resumeMd, 'resume', applicationId);
}

/**
 * Write a cover-letter Markdown to a temp DOCX file and return its absolute path.
 * The file lives in os.tmpdir() and is left for the OS to reap.
 */
export async function writeCoverLetterDocxToTemp(coverLetterMd: string, applicationId: string): Promise<string> {
  return writeDocxToTemp(coverLetterMd, 'cover-letter', applicationId);
}
```

Note: this changes the resume file's filename from `crosswalk-<id>-<ts>.docx` to `crosswalk-resume-<id>-<ts>.docx`. The existing `services.resumeFile.test.ts` first test asserts `path.includes('app-abc123')` (still true) and `path.endsWith('.docx')` (still true) — so it stays green without changes.
